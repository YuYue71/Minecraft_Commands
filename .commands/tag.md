# `/tag`（實體標籤管理指令）
* > 用於為遊戲世界中的指定實體（Entities）動態增減或查詢自訂的字串標籤（Scoreboard Tags）
* > 標籤資料直接寫入實體底層的 `Tags` NBT 陣列中，是實體分類、物件追蹤與群組化管理的最高效輕量化手段
* > 常用於區分玩家陣營、標記特定地圖機制觸發物（如特殊箭矢、隱形盔甲架）、縮減目標選擇器（Target Selectors）的比對開銷

---

## 語法結構 (Syntax)
```commands id="tag"
/tag <目標> add <標籤名稱>
/tag <目標> remove <標籤名稱>
/tag <目標> list
```

---

## 參數與引數拆解 (Arguments)
> 詳細解構語法中出現的每一個變數之填寫規範與底層資料型態

| 參數名稱 | 功能與語義說明 |
| --- | --- |
| `[必填]` `<目標>` | 指定要操作或查詢標籤的目標實體選擇器（如 `@e[type=zombie]`、`@p`） |
| `[必填]` `add` | 增加標籤關鍵字，用於將指定的自訂字串寫入目標實體的標籤列表中（若標籤已存在則自動忽略） |
| `[必填]` `remove` | 移除標籤關鍵字，用於將指定的自訂字串從目標實體的標籤列表中抹除（若標籤不存在則自動忽略） |
| `[必填]` `list` | 查詢關鍵字，用於在聊天欄列出指定目標實體當前所擁有的所有標籤清單 |
| `[必填]` `<標籤名稱>` | 自訂的識別識別碼字串，不支援包含空格、半形逗號或特殊控制字元 |

---

## 參數枚舉列表 (Parameter Enumeration)

### 子指令操作模式 (Tag Operations)

| 參數 | 說明與引擎底層行為 |
| --- | --- |
| `add` | 將該字串寫入 NBT 的 `Tags` 列表中。若目標包含多個實體（如 `@e`），會為所有符合條件的實體逐一加上標籤。 |
| `remove` | 從 NBT 的 `Tags` 列表中剔除該字串。若目標包含多個實體，會從所有符合條件的實體身上同步拔除標籤。 |
| `list` | 讀取目標的 `Tags` 欄位並格式化輸出。若選擇器對應到多個實體，引擎將預設僅列出符合條件的「第一個」實體（或報錯提示無法同時讀取多個實體）。 |

---

## 數值與字串規則

### 標籤命名格式限制 (Tag String Rules)

| 限制項目 | 規範與技術邊界說明 |
| --- | --- |
| **字元合法性** | 標籤名稱必須為嚴格的單一連續字串。**不允許包含空格、引號、半形逗號 `,**`（因為逗號在目標選擇器中被保留作為參數分隔符）。允許使用底線 `_`、橫槓 `-`、加號 `+` 或句點 `.`。 |
| **長度限制** | 雖然引擎沒有硬性對字串長度設下極低的常數上限，但建議長度保持在 32 字元以內，避免增加 NBT 序列化與資料包網路同步的頻寬負擔。 |
| **大小寫敏感度** | **嚴格區分大小寫**。例如 `Active` 與 `active` 在底層會被判定為兩個完全獨立的標籤。 |

---

## 核心限制與效能優化規範
> `/tag` 在指令系統中具備極高的存取優先級，使用時需注意以下引擎特性：

1. **零效能開銷追蹤**：
* 相較於 `/scoreboard` 必須在記憶體中維護全局的欄位矩陣，`/tag` 僅儲存於實體各自的 NBT 樹狀結構中。在目標選擇器中使用 `tag=識別碼` 進行篩選時，效率遠高於 `scores={計分項=數值}`。


2. **與 NBT 修改的本質差異**：
* 雖然標籤儲存在 NBT 中，但使用 `/tag` 變更標籤**不會**受到玩家 NBT 無法修改的限制。指令系統允許使用 `/tag` 直接對玩家實體進行 `add` 或 `remove` 操作。


3. **區塊卸載與實體死亡**：
* 非玩家實體身上的標籤會隨著實體一同儲存於區塊（Region 文件）中。當實體被殺死並從世界中移除（`Killed`）時，其附帶的標籤資料會隨之徹底銷毀。
