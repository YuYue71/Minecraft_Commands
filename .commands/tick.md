# `/tick`（遊戲刻與時間速率控制指令）
* > 用於動態調整、凍結、步進或分析遊戲引擎底層的遊戲刻（Tick）運行速率
* > 負責直接干涉全域伺服器線程的循環計時器，允許進行非同步的超速衝刺或逐影格除錯
* > 常用於複雜紅石電路偵錯、命令方塊與函數性能壓測、動畫/機制慢動作逐格檢視以及長延遲自動化測試

---

## 語法結構 (Syntax)
```commands id="tick_control"
/tick query
/tick rate <速率值>
/tick freeze
/tick unfreeze
/tick step <步進時間>
/tick step stop
```
```commands id="tick_sprint"
/tick sprint <衝刺時間>
/tick sprint stop
```

---

## 參數與引數拆解 (Arguments)
> 詳細解構語法中出現的每一個變數之填寫規範與底層資料型態

| 參數名稱 | 功能與語義說明 |
| --- | --- |
| `[必填]` `query` | 查詢關鍵字，用於回傳當前的目標 Tick 速率、運作狀態（凍結/運行）以及伺服器當前真實的每刻耗時（MSPT） |
| `[必填]` `rate` | 速率設定關鍵字，用於修改伺服器每秒嘗試運行的目標遊戲刻數量 |
| `[必填]` `freeze` | 凍結關鍵字，強行暫停除玩家實體、指令與渲染線程之外的所有世界邏輯更新（時間停止） |
| `[必填]` `unfreeze` | 解凍關鍵字，恢復被凍結的世界，使其重新按照目前的目標速率常態運作 |
| `[必填]` `step` | 步進關鍵字，在**世界處於凍結狀態下**，允許世界精確解除凍結並向後運行指定的時間長度，隨後自動重新凍結 |
| `[必填]` `sprint` | 衝刺關鍵字，解除伺服器每秒 20 次的計時器節制，以硬體極限效能全力全速向後運行指定的時間長度，常用於跳過漫長等待 |
| `[必填]` `stop` | 停止關鍵字，用於強行中斷當前正在進行中的步進（`step`）或超速衝刺（`sprint`）行為 |
| `[必填]` `<速率值>` | 浮點數值，指定每秒的目標 Tick 數（TPS） |
| `[必填]` `<步進時間>` / `<衝刺時間>` | 指定步進或衝刺的總長度，由數字搭配時間單位後綴（`t`、`s`、`d`）組成的自訂字串 |

---

## 參數枚舉列表 (Parameter Enumeration)

### 時間單位後綴 (Time Units)
> 用於 `step` 與 `sprint` 後方的時間長度指定

| 後綴符號 | 說明與轉換邏輯 |
| --- | --- |
| `t` | **遊戲刻 (Tick)**：基礎時間單位（若省略單位後綴，預設以此單位解析）。 |
| `s` | **實體秒 (Seconds)**：以秒為單位計算。引擎會將其數值乘以目前定義的目標速率自動轉化為總遊戲刻。 |
| `d` | **遊戲天 (Days)**：以 24000 遊戲刻為一基準天進行折算。 |

---

## 數值與邊界規則

### 1. 速率邊界限制 (Rate Boundaries)

| 參數 | 說明 |
| --- | --- |
| `<速率值>` | 最小值：`1.0`（極限慢動作，每秒僅運算 1 刻） / 最大值：`10000.0`（極限超速，每秒嘗試運算一萬刻） / 預設值：`20.0`（即標準 20 TPS，每刻耗時 50ms） |

### 2. 衝刺與步進邊界 (Sprint / Step Range)

| 參數 | 說明 |
| --- | --- |
| `<時間>` | 最小值：`1t` / 最大值：無嚴格理論上限（受 32 位元有號整數有上限 `2147483647t` 制約） |

---

## 核心限制與控制流運作規範
> `/tick` 是直接作用於伺服器主循環（Server Main Loop）的最高權限控制，其運行時具備以下底層特性：

1. **凍結邊界（What Freezes?）**：
* 當執行 `/tick freeze` 時：**實體 AI、方塊刻（如作物生長、隨機刻）、紅石更新、天氣流逝、晝夜更替、液體流動**將全部暫停。
* **不受影響的項目**：玩家本體可以自由移動與互動（除非被單獨限制）、網路封包收發正常、客戶端粒子渲染正常、`/execute` 或命令方塊指令仍能正常按 Tick 執行。


2. **衝刺行為與硬體瓶頸（Sprint Mechanics）**：
* 當執行 `/tick sprint <時間>` 時，伺服器會徹底關閉 `Thread.sleep()` 的節流計時，以單核 CPU 100% 的運算開銷瘋狂執行世界刻更新。
* **注意**：如果在衝刺期間伺服器底層的真實 MSPT（每刻耗時）大於 50ms（遭遇卡頓），衝刺速度將受限於硬體極限，此時客戶端與伺服器可能會產生短暫的非同步拉回現象。


3. **互斥邏輯**：
* `sprint` 與 `freeze` 為互斥行為。在世界被凍結時執行 `sprint` 會自動解除凍結以進行超速運算，衝刺結束後世界將保持正常運行狀態（不會自動回撥為凍結）。
