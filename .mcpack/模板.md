## [路由層] 格式分流規則 (Format Router)

在生成任何內容之前, 必須先執行格式判定, 根據目標元素的檔案格式選擇對應的正文框架.

### 判定流程

```
輸入目標元素
    │
    ├─ 副檔名為 .mcfunction ? ─── 是 ──→ 套用 [框架 B: mcfunction 框架]
    │
    ├─ 副檔名為 .nbt ?       ─── 是 ──→ 套用 [框架 C: nbt 框架]
    │
    └─ 副檔名為 .json ?      ─── 是 ──→ 進入 [結構複雜度判定] ──→ 套用 [框架 A: json 框架]
```

### 結構複雜度判定 (僅適用 .json 元素)

判定後決定框架 A 內哪些區塊需要生成、哪些可省略.

```
根節點鍵值數量 ≤ 6 且無嵌套子對象
    └─ 扁平元素 ──→ 省略 §4 子對象詳解, 省略 §6 數據約束 (無數值類型時)

根節點存在 "type" 鍵值且其值決定其餘結構
    └─ 多型態元素 ──→ §2 核心結構範本須按 type 分支逐一列出骨架
                  ──→ §4 子對象詳解須為每個 type 分支建立獨立 H3 小節

根節點嵌套深度 ≥ 3 層
    └─ 深層嵌套元素 ──→ §4 子對象詳解須遞迴展開至最底層葉節點
                    ──→ 每層以獨立 H3 小節呈現, 層級關係用標題編號體現 (如 4.1, 4.1.1)
```

## [框架 A] JSON 元素正文框架

適用元素: `loot_tables`, `advancements`, `recipes`, `tags`, `predicates`, `item_modifiers`, `damage_type`, `chat_type`, `dimension`, `dimension_type`

正文框架即本模板上方的 §1 - §7 標準區塊結構. 依結構複雜度判定結果決定各區塊的展開深度.

### 區塊生成規則

**§1 目錄路徑** — 無條件生成. 副檔名固定為 `.json`.

**§2 核心結構範本**
- 扁平元素: 單一骨架即可.
- 多型態元素: 按 `type` 分支逐一輸出骨架, 每個骨架前加注釋標明分支名稱.

  範例格式:
  ```json
  // type: "minecraft:crafting_shaped"
  {
    "type": "minecraft:crafting_shaped",
    "pattern": [],
    "key": {},
    "result": {}
  }

  // type: "minecraft:smelting"
  {
    "type": "minecraft:smelting",
    "ingredient": {},
    "result": "",
    "experience": 0,
    "cookingtime": 200
  }
  ```

**§3 根節點屬性表**
- 多型態元素: 表格首行為 `"type"` 欄位並索引至 §5 枚舉. 其餘欄位依所屬分支在說明欄標注適用範圍 (例如: `僅適用 crafting_shaped`).
- 扁平元素: 直接列出所有鍵值, 無需分支標注.

**§4 子對象詳解**
- 扁平元素 (根節點鍵值數量 ≤ 6 且無嵌套): 省略此區塊.
- 有嵌套子對象: 每個獨立子對象建立一個 H3 小節. 多型態元素中不同分支的子對象各自建立小節, 標題標明所屬分支.
- 深層嵌套: 遞迴展開, 以 `4.1`, `4.1.1`, `4.1.2` 等編號體現層級.

**§5 參數枚舉與可用值全清單**
- 無枚舉欄位 (如 `tags`): 省略此區塊.
- 有枚舉欄位: 每個枚舉欄位建立一個 H3 小節. 枚舉值必須窮舉完整, 禁止省略.
- 多型態元素的 `type` 枚舉: 建立獨立 H3 小節, 每個值的說明欄標注該分支的核心用途.

**§6 數據約束與邊界條件**
- 無數值類型且無陣列長度限制 (如 `tags`, `chat_type`): 省略此區塊.
- 有數值類型: 列出所有 `int` / `float` 欄位的硬性範圍與違規後果.
- 有陣列: 列出最小長度限制.

**§7 外部連結** — 無條件生成.

## [靜態規則層] 全域生成規範

以下規則適用於所有框架, 優先級低於路由層判定結果, 但高於 AI 的自由裁量.

### 規範 1: [絕對約束] 標點符號與格式 (Absolute Constraints)

* 整份文件嚴禁使用全形標點符號 (如: ， 。 ： ； （ ）). 所有標點符號一律強制替換為半形 (如: , . : ; ( )).
* 輸出純 Markdown 格式, 嚴禁附加任何提示語, 說明文字或對話. 所有佔位符必須完全替換.

### 規範 2: 標頭與目錄規則 (Header & TOC Rules)

* 標頭版本統一維持 `JE ≤ 1.20.4`.
* 目錄 (TOC) 的連結文字必須使用中文, 連結目的地統一填入 `#`, 不填寫實際錨點 ID.
* 所有 H2/H3 標題後方嚴禁附加 `{#anchor-id}` 語法.
* 目錄條目必須與正文實際生成的區塊完全對應: 省略的區塊必須同步從目錄移除, 新增的區塊必須同步補入目錄.

### 規範 3: 語法與參數表規則 (Syntax & Parameter Table Rules)

* 表格標頭必須嚴格使用 `| 鍵值 (Key) | 類型 | 必填 | 簡述 | 說明、預設值與可用值索引 |`.
* 必填欄位的預設值填 `-`, 不可留空或填 `X`.
* 目標選擇器 (Target Selectors) 與常規座標 (Coordinates) 強制不展開說明, 僅保留於全域語法列. 若牽涉體積運算 (如 dx, dy, dz) 則必須列出體積參數.

### 規範 4: 子元素與枚舉規則 (Subcommand & Enum Rules)

* 枚舉資料 (Enum Data) 必須查找完整並全部列出, 絕對禁止使用「... 等」或「其餘略」進行省略. 無法確認的說明填 `-`.
* 若某參數具備預設值, 必須在枚舉表格的說明欄位內明確標註 `(預設值)`.
* 已棄用 (Deprecated) 的枚舉值仍須列出, 並於說明欄標註「已棄用 Deprecated」.

### 規範 5: 長文件分段規則 (Pagination Rules)

* 若預估枚舉條目過多 (如音效, 粒子, 物品等), 導致單次 Token 輸出可能超載, 必須先輸出主體骨架與表頭, 並使用 `<!-- PENDING -->` 標註, 待確認後再進行後續分段填補.

### 規範 6: 全域系統自動映射與去冗餘規則 (Cross-Reference Auto-Linking & No-Redundancy)

* 只要該元素的語法或參數中, 有使用或提及下列任何全域系統機制, 必須強制將對應的 Markdown 連結直接複製並加入到 `## 外部連結 (References)` 區塊中.

### 規範 7: 無外部組件映射之窮舉處理原則 (Inline Exhaustive List Fallback)

* 觸發條件: 當元素包含需要列舉的特定 ID, 參數或資料清單 (如進度 ID, 遊戲規則等), 且該項目不在規範 6 的外部連結清單中時.
* 處理動作: 絕對不可要求讀者自行去 Wiki 查詢或僅提供零星範例. 必須直接在速查表的 `參數說明` 下方, 新增一個獨立區塊 (例如 `## 原版 [項目名稱] 清單`).
* 排版要求: 必須將所有原版資料使用高密度的「分類標題 (###)」與「Markdown 矩陣表格」進行歸納排版, 確保清單完整, 直觀且無冗餘贅字.

### 規範 8: 管道符號跳脫原則 (Pipe Character Escaping)

* 觸發條件: 當需要在 Markdown 表格的儲存格內部顯示垂直線 `|` 時 (例如表示二擇一的參數 `[A|B]`).
* 處理動作: 必須強制使用反斜線進行跳脫, 寫成 `\|` (例如 `[A\|B]`), 絕對不可直接輸出純 `|`, 以防止表格渲染結構損毀.

### 規範 9: 資料包引用與結構說明原則 (Datapack Reference Rules)

* 觸發條件: 當元素的參數涉及依賴資料包 (Datapack) 定義的動態資源 (如 Loot Tables, Recipes, Advancements, Predicates, Item Modifiers, Functions 等) 時.
* 處理動作:
  1. 必須在參數表的「說明」欄位內明確標註「需透過資料包定義」.
  2. 必須在速查表內部 (`## 外部連結` 的上方), 新增一個獨立的 `## 資料包結構參考 (Datapack Structure)` 區塊.
* 排版要求: 在該獨立區塊內, 必須列出該資源在資料包中的「標準目錄路徑」 (例如 `data/<namespace>/loot_tables/`), 並提供其 JSON 檔案的核心結構範例或關鍵節點說明, 以確保讀者能確實理解如何建立該資源.

### 規範 10: 雙重類型欄位標示原則 (Polymorphic Field Notation)

* 觸發條件: 當某欄位同時支援兩種以上的合法輸入格式時 (如 `float` 單值或 `[float, float]` 區間陣列).
* 處理動作: 類型欄位內使用 `\|` 分隔各合法格式 (例如 `float \| [float, float]`). 說明欄位內必須逐一描述每種格式的語意差異與適用情境. 不可僅標注其中一種格式而忽略另一種.

### 規範 11: 目錄動態同步原則 (TOC Sync Rule)

* 觸發條件: 每次依據結構複雜度判定省略或新增任何正文區塊時.
* 處理動作: 目錄必須與實際生成的正文區塊完全對應. 省略的區塊從目錄移除; 新增的非標準區塊 (如 `## 原版清單`, `## 資料包結構參考`) 必須補入目錄. 禁止目錄與正文不一致.
